# 阶段三文档冻结设计说明

> 日期：2026-04-14
> 复杂度：large
> 状态：已完成

## 背景

当前仓库中的第三阶段文档仍基于较早方案，和已经拍板的新方向存在明显偏差：

- 实时通知仍混有轮询方案，但课程要求与当前决策要求 `WebSocket`
- 信用体系仍以规则型为主，但当前决策改为双分制加权模型
- 架构仍以阿里云函数计算 FC 为主，但当前决策改为单云主机/容器主部署
- 管理端仍偏 MVP，但当前决策改为完整运营后台 + 首页内容配置
- 进阶功能尚未完整纳入 IM、地图、DeepSeek 审核、智能推荐
- Git 协作仍是旧的 `develop` 思路，当前决策改为 `main / check / feat/*`

## 目标

- 统一冻结阶段三文档基线，确保 PRD、架构、数据库、API、分工、Git 六份文档一致
- 把所有会影响开发与联调的关键决策写死，减少中途返工
- 让组长可以在文档更新后直接给组员派活开工

## 非目标

- 不实现业务代码
- 不补充未拍板的新功能
- 不在本轮引入真实导航路径规划或复杂机器学习推荐

## 冻结基线

- 必做：任务状态机、虚拟账户与资金、实时通知、信用评价
- 进阶：任务内 IM、北邮楼宇级校园地图、DeepSeek 审核、智能推荐
- 实时能力：全部以 `WebSocket` 为基线
- 部署：单云主机/容器主部署，包含前端静态站点、FastAPI、MySQL、Redis、Nginx
- 信用：双分制 + 加权模型 + 后台可配置权重
- 管理端：完整运营后台 + 首页内容配置
- Git：`main / check / feat/*`
- 冻结治理：组长单点拍板，其他成员通过 RFC/PR 提建议
