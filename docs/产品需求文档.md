# CampusMast 产品需求文档（PRD）

> 版本：v2.0
> 更新日期：2026-04-14
> 状态：冻结版本

---

## 1. 项目背景

CampusMast 是一个面向北京邮电大学校园场景的互助与众包任务平台。学生可以发布代取快递、代买餐食、搬运重物等即时需求，其他学生可以接单完成任务并获得报酬。平台的核心目标是通过任务状态机、资金托管、实时消息、信用评价和运营后台，构建一个可闭环运行的校园服务平台。

本阶段文档冻结后，开发必须围绕以下基线展开：

- 必做模块：任务全生命周期管理、虚拟账户与资金系统、实时通知系统、信用与评价体系
- 进阶模块：任务内 IM、北邮楼宇级校园地图、DeepSeek AI 审核、智能任务推荐
- 技术基线：`WebSocket` 实时通信、双分制加权信用分、完整运营后台、云主机/容器主部署

## 2. 目标用户

| 用户群体 | 典型场景 |
|---------|---------|
| 需求方（Requester） | 发布任务、支付赏金、验收结果、查看地图位置、与接单方沟通 |
| 接单方（Helper） | 浏览任务、按地图距离筛选、接单执行、提交证明、获取报酬 |
| 管理员（Admin） | 管理用户、处理争议、管理 AI 审核、管理通知模板、配置运营规则、维护首页内容 |

## 3. 角色与权限定义（RBAC）

### 3.1 需求方（Requester）

| 权限 | 说明 |
|------|------|
| 发布任务 | 填写任务信息、上传图片、选择校园楼宇点位、设置报酬与截止时间 |
| 取消任务 | 仅限 `PENDING` 状态，取消后资金自动解冻 |
| 验收任务 | 接单方提交完成证明后确认或拒绝 |
| 发起争议 | 对完成结果不满意时发起争议，进入管理员处理流程 |
| 与接单方聊天 | 任务接单后可通过任务内 IM 单聊 |
| 查看推荐与地图 | 查看任务推荐结果、地图点位、附近任务 |
| 钱包操作 | 模拟充值、提现、查看流水 |
| 评价接单方 | 任务完成后给接单方评分与评论 |

### 3.2 接单方（Helper）

| 权限 | 说明 |
|------|------|
| 浏览任务池 | 查看所有 `PENDING` 任务，支持地图、分类、关键词、报酬与距离筛选 |
| 接单 | 满足信用门槛时锁定任务，进入 `IN_PROGRESS` |
| 提交完成 | 上传图片/文字完成证明 |
| 放弃任务 | 从 `IN_PROGRESS` 回退到 `PENDING`，影响信用分 |
| 与需求方聊天 | 在任务内实时沟通 |
| 查看信用与推荐 | 查看自身双信用分与推荐任务 |
| 钱包操作 | 查看余额、流水、自动收款记录 |
| 评价需求方 | 任务完成后给需求方评分与评论 |

### 3.3 管理员（Admin）

| 权限 | 说明 |
|------|------|
| 用户管理 | 查看用户、封禁/解封、调整双信用分 |
| 任务管理 | 查看所有任务、查看地图任务分布、处理违规任务 |
| 争议裁决 | 裁定退款、结算、拆分结算或管理员关闭 |
| AI 审核管理 | 查看审核记录、复审低危内容、维护审核阈值 |
| 通知与消息管理 | 管理通知模板、查看消息统计 |
| 运营规则配置 | 配置接单门槛、信用权重、推荐参数、审核阈值、任务分类 |
| 首页内容配置 | 配置公告、Banner、推荐位 |
| 数据统计 | 查看用户、任务、消息、审核、交易、地图热度等统计数据 |

> **说明**：
> - 同一用户可在不同任务中同时充当需求方和接单方。
> - 用户不能接自己发布的任务。
> - 接单资格以 `helperCreditScore >= 60` 为默认门槛。

## 4. 核心业务模块

### 4.1 用户认证与资料

#### 4.1.1 注册

- 使用学校邮箱注册（如 `xxx@bupt.edu.cn`）
- 填写邮箱、密码、昵称
- 注册成功后自动创建钱包
- 初始化三项信用分：
  - `requesterCreditScore = 100`
  - `helperCreditScore = 100`
  - `overallCreditScore = 100`

#### 4.1.2 登录

- 邮箱 + 密码登录
- 返回：
  - `accessToken`
  - `refreshToken`
  - 当前用户基础信息
- 支持刷新令牌静默续期

#### 4.1.3 个人资料

- 可编辑：昵称、手机号、头像、常用楼宇
- 只读展示：
  - `requesterCreditScore`
  - `helperCreditScore`
  - `overallCreditScore`
  - 已完成任务数
  - 平均评分

### 4.2 任务全生命周期管理（核心）

#### 4.2.1 发布任务

| 字段 | 类型 | 必填 | 说明 |
|------|------|------|------|
| title | string | 是 | 标题，2-50 字 |
| description | string | 是 | 描述，10-500 字 |
| category | enum | 是 | `package` / `food` / `move` / `other` |
| reward | decimal | 是 | 报酬金额，最小 1.00 |
| deadline | datetime | 是 | 必须晚于当前时间 |
| buildingCode | string | 是 | 校园楼宇编码 |
| locationDetail | string | 否 | 补充地点描述 |
| imageUrls | string[] | 否 | 最多 3 张 |

**发布前同步审核规则**

- 先执行本地敏感词预检；命中敏感词时直接拒发，不创建任务、不冻结资金，但写入审核记录
- 未命中敏感词时调用 DeepSeek 审核任务标题、描述与图片说明
- DeepSeek 返回 `accept` 时直接发布；返回 `reject` 时拒发；返回 `refuse` 时允许发布，但自动标记为“待管理员复审”
- `refuse` 必须携带 `LOW` / `MEDIUM` / `HIGH` 风险提示，审核记录需保留 AI 结果、风险提示和原因
- DeepSeek 未配置或调用异常时按 `refuse` + `MEDIUM` 兜底进入复审，避免直接放行
- 人工复审通过时任务继续展示；人工复审拒绝时，仍处于 `PENDING` 的任务下架为 `CLOSED_BY_ADMIN`，解冻赏金并写入任务日志
- 用户可以在自己的审核记录中看到 AI 审核、风险提示、人工复审结论和理由

**资金规则**

- 发布任务时自动冻结赏金
- 冻结成功后任务进入 `PENDING`
- 余额不足时发布失败

#### 4.2.2 任务状态机

任务状态与流转固定如下：

```text
PENDING -> IN_PROGRESS -> PENDING_REVIEW -> COMPLETED
PENDING -> CANCELLED
PENDING -> EXPIRED
PENDING -> CLOSED_BY_ADMIN
IN_PROGRESS -> PENDING
IN_PROGRESS -> DISPUTED
PENDING_REVIEW -> DISPUTED
DISPUTED -> COMPLETED
DISPUTED -> CANCELLED
DISPUTED -> CLOSED_BY_ADMIN
```

#### 4.2.3 状态转换规则

| 当前状态 | 目标状态 | 触发角色 | 条件 | 附带动作 |
|---------|---------|---------|------|---------|
| PENDING | IN_PROGRESS | Helper | `helperCreditScore >= 60` 且未被他人接单 | 写入 `helperId`、发通知、开启聊天会话 |
| PENDING | CANCELLED | Requester | 任务未被接单 | 解冻资金、写日志、发通知 |
| PENDING | EXPIRED | System | 超过截止时间 | 解冻资金、写日志、发通知 |
| PENDING | CLOSED_BY_ADMIN | Admin | 人工复审拒绝或运营下架 | 解冻资金、写日志、记录复审理由 |
| IN_PROGRESS | PENDING_REVIEW | Helper | 提交完成证明 | 写入证明、发通知 |
| IN_PROGRESS | PENDING | Helper | 主动放弃 | 清空 `helperId`、更新信用、发通知 |
| IN_PROGRESS | DISPUTED | Requester | 举报或说明服务异常 | 记录争议、通知管理员 |
| PENDING_REVIEW | COMPLETED | Requester | 验收通过 | 结算资金、更新信用、允许互评 |
| PENDING_REVIEW | DISPUTED | Requester | 拒绝验收 | 记录争议、通知管理员 |
| DISPUTED | COMPLETED | Admin | 支持接单方 | 结算资金、更新信用 |
| DISPUTED | CANCELLED | Admin | 支持需求方 | 退款、更新信用 |
| DISPUTED | CLOSED_BY_ADMIN | Admin | 自定义裁决 | 可拆分结算、记录备注 |

#### 4.2.4 任务大厅

- 默认展示全部 `PENDING` 任务
- 支持分类、关键词、报酬区间、楼宇、距离排序
- 距离排序优先使用用户选点坐标 `userLatitude` / `userLongitude`；未传选点时可退回 `nearBuildingCode` 楼宇中心
- 列表任务需要返回 `distanceMeters`，用于展示任务到当前选点或楼宇原点的具体距离
- 支持“推荐任务”入口；推荐排序需要返回类别、距离、履约成功率、活跃时间四项得分和推荐原因
- 支持地图模式与列表模式切换

#### 4.2.5 任务详情

- 展示：
  - 基础信息
  - 发布者/接单方信息
  - 地图楼宇位置
  - 状态时间线
  - 聊天入口
  - AI 审核摘要（用户仅看到安全提示，管理员看到完整记录）
- 按角色与状态展示不同操作按钮

#### 4.2.6 我的任务

- Tab 切换：我发布的 / 我接的
- 支持按状态筛选
- 支持查看聊天未读、地图位置、评价状态

### 4.3 虚拟账户与资金系统（核心）

#### 4.3.1 钱包模型

每个用户一个钱包，包含：

| 字段 | 说明 |
|------|------|
| available | 可用余额 |
| frozen | 冻结余额 |

总余额 = `available + frozen`

#### 4.3.2 资金操作

| 操作 | 场景 | available 变化 | frozen 变化 | 流水类型 |
|------|------|---------------|------------|---------|
| 充值 | 模拟充值 | `+amount` | 不变 | `TOP_UP` |
| 提现 | 模拟提现 | `-amount` | 不变 | `WITHDRAW` |
| 冻结 | 发布任务 | `-reward` | `+reward` | `FREEZE` |
| 解冻 | 取消/过期/退款 | `+reward` | `-reward` | `UNFREEZE` |
| 结算支出 | 任务完成 | 不变 | `-reward` | `SETTLE_OUT` |
| 结算收入 | 任务完成 | `+reward` | 不变 | `SETTLE_IN` |
| 拆分结算 | 争议裁决 | 按裁决比例变化 | 按裁决比例变化 | `SETTLE_SPLIT` |

#### 4.3.3 一致性要求

- 资金冻结、解冻、结算必须放在同一事务中
- 接单、完成、争议裁决与资金操作必须在同一业务事务内联动
- 不允许 `available` 或 `frozen` 为负
- 并发接单只能有一人成功

### 4.4 信用与评价体系（核心）

#### 4.4.1 双分制信用模型

系统维护三项分数：

| 字段 | 说明 |
|------|------|
| requesterCreditScore | 需求方信用分 |
| helperCreditScore | 接单方信用分 |
| overallCreditScore | 综合展示信用分，默认 `helper 60% + requester 40%` |

默认范围：`0-100`

#### 4.4.2 Helper 加权模型

| 指标 | 权重 |
|------|------|
| 完成率 | 35% |
| 平均评分 | 25% |
| 超时率反向得分 | 15% |
| 放弃率反向得分 | 15% |
| 争议败诉率反向得分 | 10% |

#### 4.4.3 Requester 加权模型

| 指标 | 权重 |
|------|------|
| 完成确认率 | 35% |
| 平均评分 | 25% |
| 超时确认率反向得分 | 15% |
| 恶意争议率反向得分 | 15% |
| 接单后取消率反向得分 | 10% |

#### 4.4.4 业务规则

- 新用户三项信用分均为 100
- 接单门槛默认为 `helperCreditScore >= 60`
- 管理后台可调整权重与门槛
- 每个完成任务支持双向互评
- 评价内容：1-5 星评分 + 评论
- 评价结果参与下一轮信用分计算

### 4.5 实时通知系统（核心）

#### 4.5.1 方案要求

- 使用 `WebSocket` 作为实时通知基线
- 不再以轮询作为主方案
- 普通列表查询仍保留 HTTP 接口

#### 4.5.2 通知事件

| 事件名 | 触发时机 | 接收方 |
|------|------|------|
| TASK_ACCEPTED | 任务被接单 | Requester |
| TASK_SUBMITTED | 提交完成证明 | Requester |
| TASK_CONFIRMED | 需求方确认完成 | Helper |
| TASK_REJECTED | 拒绝验收 | Helper |
| TASK_CANCELLED | 任务取消/过期 | 相关方 |
| TASK_DISPUTED | 进入争议 | 双方 + Admin |
| DISPUTE_RESOLVED | 争议裁决完成 | 双方 |
| CHAT_MESSAGE | 收到新聊天消息 | 任务聊天参与方 |
| CHAT_READ | 对方已读消息 | 任务聊天参与方 |
| MODERATION_REVIEW | 低危内容进入复审 | Admin |
| SYSTEM_NOTICE | 系统公告更新 | 全体或指定用户 |

#### 4.5.3 用户侧能力

- 导航栏未读角标
- 通知中心分页列表
- 单条已读 / 全部已读
- WebSocket 断线重连

## 5. 进阶功能

### 5.1 任务内 IM

- 每个接单成功的任务自动创建一个聊天会话
- 支持文本消息、离线未读、WebSocket 实时推送
- 暂不支持群聊与音视频

### 5.2 北邮校园地图

- 使用 OpenStreetMap 真实地图底图
- 按楼宇维度维护 `buildingCode`
- 前端基于楼宇中心经纬度和可选楼宇多边形展示地图覆盖层
- 支持展示任务点位、用户常用楼宇、附近任务排序
- 首版不做路径导航，只做距离估算

### 5.3 DeepSeek AI 审核

- 发布任务前同步审核
- 高危内容直接拦截：暴力、色情、诈骗、违法
- 低危内容允许提交但进入管理员复审：辱骂、人身攻击、广告刷屏
- 审核结果进入后台审核管理

### 5.4 智能任务推荐

- 默认信号：
  - 任务类别偏好
  - 楼宇距离
  - 历史完成成功率
  - 近期活跃时段
- 首版采用规则加权推荐，不使用复杂机器学习

## 7. 非功能需求

### 7.1 安全性

- 密码使用 bcrypt 哈希
- JWT 保护除注册/登录外的接口
- 聊天与通知 WebSocket 连接需鉴权
- 任务发布前执行内容审核

### 7.2 性能

- 列表统一分页
- 地图附近任务查询需支持楼宇和距离索引
- Redis 用于未读计数、会话缓存、推荐缓存

### 7.3 可用性

- 移动端优先
- WebSocket 断线重连
- 关键操作有 loading、错误反馈、空状态页

### 7.4 一致性

- 钱包事务与状态机联动一致
- 并发接单仅一人成功
- 消息已读状态与未读角标一致

## 8. 开发验收要点

开发与联调阶段必须覆盖以下场景：

- 单任务完整生命周期流转
- 多人并发接单，仅一人成功
- 钱包冻结、解冻、结算一致
- WebSocket 通知即时送达
- 任务内 IM 文本消息实时收发与离线补收
- 地图附近任务排序正确
- DeepSeek 高危内容拦截、低危内容复审
- 推荐结果能够体现类别偏好与距离权重
- 管理后台规则调整后即时生效
