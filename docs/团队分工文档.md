> 状态：冻结版本

## 1. 分工总原则

本项目采用“**组长单点拍板 + 模块独占开发 + `check` 统一集成**”模式，目标是：

- 避免接口、状态机、页面结构在开发中反复变动
- 避免多人同时改同一类核心文件造成高耦合
- 保持前后端并行推进，但所有关键基线只有一个决策出口

冻结后的协作原则如下：

- 组长负责文档、原型、接口、状态机、错误码、事件枚举、地图点位、审核规则的最终拍板
- A 负责前端工程实现，不负责重新定义页面结构
- B 负责任务状态机、钱包事务、信用算法、评价与推荐逻辑
- 所有开发分支统一向 `check` 提交 PR
- 仅组长负责 review 并合并到 `check`

## 2. 角色定义

### 2.1 组长 / C 同学

**职责定位**

- 项目总体负责人
- 原 C 角色能力负责人
- UI 原型与冻结治理负责人

**负责内容**

- 静态 HTML 原型与交互稿
- PRD、系统架构、数据库、API 文档最终拍板
- 后端基础设施
- 认证与鉴权
- WebSocket 通知与 IM 基础设施
- 管理后台
- 部署与最终集成

**不负责的内容**

- 不负责将静态原型全部 Vue 化落地
- 不负责任务状态机、钱包、信用算法的主实现

### 2.2 A 同学

**职责定位**

- 前端工程负责人

**负责内容**

- 将组长提供的静态 HTML 原型实现为 Vue 页面/组件
- 前端工程化
- 路由、状态管理、接口封装、本地后端联调
- WebSocket 客户端接入
- 地图页面前端展示

**边界说明**

- 可调整样式、响应式细节、组件拆分方式
- 不得擅自修改页面结构、交互流程、字段含义、接口字段名
- 如发现原型或接口存在问题，先提 RFC，由组长拍板

### 2.3 B 同学

**职责定位**

- 核心业务负责人

**负责内容**

- 任务状态机实现
- 钱包冻结、解冻、结算事务
- 双分制加权信用模型
- 互评系统
- 智能任务推荐逻辑
- 核心测试与并发场景验证

**边界说明**

- 不负责认证、后台系统配置页面
- 若状态机、信用字段、推荐特征需要变更，必须先更新冻结文档

## 5. 高耦合风险与规避方式

### 5.1 主要高耦合点

| 风险点 | 说明 | 规避方式 |
|------|------|------|
| 页面结构与接口字段双向修改 | A 实现时自行改字段/结构会导致接口返工 | 页面结构与字段由组长冻结，A 只做实现层调整 |
| 状态机与钱包事务耦合 | 任务流转和资金处理必须一起变更 | 由 B 统一负责任务+钱包主逻辑 |
| WebSocket 与 IM/通知耦合 | 事件名和载荷不统一会导致前后端返工 | 事件枚举与载荷由组长统一维护在 API 文档 |
| AI 审核与任务发布耦合 | 审核结果影响发布接口返回与后台流程 | 审核规则、返回码、后台复审入口一次性冻结 |
| 推荐与地图/信用耦合 | 距离特征和信用特征如果不稳定会导致推荐接口反复改 | 推荐特征固定为四项，先不扩张 |

### 5.2 冻结文件不再“三人共同维护”

以下内容不允许三人共同直接修改：

- `API接口规范.yaml`
- 状态机枚举
- 错误码
- WebSocket 事件名
- 信用分字段
- 地图楼宇点位字段
- 审核结果枚举

这些内容统一由组长维护，其他成员通过 RFC 或 PR 提建议。

## 6. 冻结治理流程

### 6.1 冻结对象

- PRD
- API 接口规范
- 数据模型
- 静态 HTML 原型
- WebSocket 事件契约
- 错误码与枚举

### 6.2 变更流程

1. 提出者在群里发 RFC
2. 说明变更原因、影响范围、建议方案
3. 组长判断是否接受
4. 如接受，先改文档，再改代码
5. 禁止先改代码后补文档

### 6.3 RFC 模板

```text
[RFC] 接口/状态/字段变更申请
模块：任务 / 钱包 / 聊天 / 地图 / 审核 / 推荐 / 后台
当前定义：
建议修改：
影响范围：
是否阻塞开发：
```

## 7. 协作顺序

### 7.1 开工前顺序

1. 组长冻结文档
2. 组长输出静态 HTML 原型
3. 组长在 `check` 分支提交冻结文档
4. A、B 从 `check` 拉出自己的 `feat/*` 分支

### 7.2 开发顺序

1. 组长先提供认证、WebSocket 基础设施、后台框架
2. B 在此基础上实现任务状态机、钱包、信用、推荐
3. A 基于页面基线实现页面，并直接连接本地后端接口开发
4. 所有人完成后向 `check` 提交 PR
5. 组长统一 review、联调、合并

## 8. 工作量平衡说明

### 8.1 组长 / C 同学

- 工作量最大，但与组长职责匹配
- 额外承担 UI 原型与文档拍板权
- 通过把前端页面实现交给 A、核心业务交给 B 来降低过载

### 8.2 A 同学

- 不再承担 UI 设计拍板
- 重点放在前端工程实现与联调
- 页面数量多，但复用度高，工作量合理

### 8.3 B 同学

- 模块少但技术深度大
- 负责事务一致性、状态机、信用模型、推荐逻辑，技术难度高

## 9. 开工验收标准

分工文档冻结后，应满足：

- 不存在“共同维护核心冻结文件”的模糊表述
- 每个关键模块都有唯一 owner
- 组长拍板权、A 的实现边界、B 的业务边界清晰
- 开发顺序与 Git 流程一致
