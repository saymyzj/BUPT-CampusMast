> 版本：v2.0
> 更新日期：2026-04-14
> 状态：冻结版本

## 1. 系统架构总览

本项目采用**前后端分离 + 单云主机/容器主部署**架构。核心组件固定为：

- 前端静态站点（Vue 3）
- FastAPI 应用
- MySQL
- Redis
- Nginx
- 阿里云 OSS
- DeepSeek API

### 1.1 逻辑架构

```text
用户浏览器
    │
    ├── HTTPS 静态资源请求
    │        ↓
    │     Nginx / 静态站点
    │        ↓
    │     Vue 3 SPA
    │
    ├── HTTPS API 请求
    │        ↓
    │     Nginx 反向代理
    │        ↓
    │     FastAPI 应用容器
    │        ├── MySQL（业务主库）
    │        ├── Redis（会话、未读、推送、推荐缓存）
    │        ├── 阿里云 OSS（图片与静态资源）
    │        └── DeepSeek API（内容审核）
    │
    └── WebSocket 长连接
             ↓
          Nginx
             ↓
          FastAPI WebSocket 服务
             ↓
          Redis 发布/订阅与未读同步
```

### 1.2 关键结论

- 不再以阿里云函数计算 FC 作为主部署方案
- `WebSocket` 为通知与 IM 的唯一实时通道
- Redis 为实时能力的必选中间件
- 地图采用 `Leaflet + OpenStreetMap` 瓦片底图与楼宇经纬度覆盖层，不依赖真实路径导航服务

## 3. 模块分层架构

```text
请求/事件
  -> Router / WebSocket Gateway
  -> Dependencies（认证、鉴权）
  -> Service（业务逻辑）
  -> Repository / ORM
  -> MySQL / Redis / OSS / DeepSeek
```

### 3.1 分层职责

| 层级 | 目录建议 | 职责 |
|------|------|------|
| Routers | `app/routers/` | HTTP API 路由定义 |
| WebSocket Gateway | `app/websockets/` | 通知与聊天事件连接、鉴权、广播 |
| Dependencies | `app/dependencies/` | 获取当前用户、管理员鉴权、数据库会话 |
| Services | `app/services/` | 状态机、钱包、信用、推荐、审核、IM、后台业务 |
| Models | `app/models/` | SQLAlchemy 模型 |
| Schemas | `app/schemas/` | API 请求/响应模型与 WebSocket 载荷模型 |
| Utils | `app/utils/` | JWT、DeepSeek、OSS、坐标工具、缓存工具 |

### 3.2 关键服务拆分

| 服务 | 说明 |
|------|------|
| `task_service` | 任务生命周期与状态机 |
| `wallet_service` | 冻结、解冻、结算、流水 |
| `credit_service` | 双分制加权信用模型计算 |
| `notification_service` | 站内通知、模板管理、实时推送 |
| `chat_service` | 任务内 IM、会话、未读、已读 |
| `map_service` | 校园楼宇点位、距离计算、附近任务查询 |
| `moderation_service` | DeepSeek 同步审核、低危复审 |
| `recommendation_service` | 任务推荐计算与缓存 |
| `admin_service` | 完整运营后台能力 |
| `config_service` | 信用权重、审核阈值、首页内容等系统配置 |

## 4. 请求与事件链路

### 4.1 普通 HTTP 请求

```text
浏览器 -> Nginx -> FastAPI Router -> Service -> MySQL/Redis/OSS/DeepSeek -> 响应
```

### 4.2 WebSocket 链路

```text
浏览器 -> Nginx -> FastAPI WebSocket -> Redis 发布/订阅 -> 目标用户连接
```

### 4.3 任务发布链路

```text
前端提交任务
  -> FastAPI 校验参数
  -> moderation_service 调用 DeepSeek 同步审核
  -> 高危则拦截返回
  -> 低危则创建任务并标记待复审
  -> wallet_service 冻结资金
  -> task_service 创建任务
  -> 写入 task_logs / moderation_records
  -> 返回任务详情
```

### 4.4 接单链路

```text
Helper 点击接单
  -> task_service 行锁/事务锁定任务
  -> 校验 helperCreditScore >= 60
  -> 更新状态为 IN_PROGRESS
  -> chat_service 创建任务聊天会话
  -> notification_service 推送 TASK_ACCEPTED
  -> 返回最新任务状态
```

### 4.5 消息链路

```text
用户发送消息
  -> WebSocket Gateway 鉴权
  -> chat_service 持久化消息
  -> Redis 发布事件
  -> 接收方在线则实时推送
  -> 接收方离线则累加未读
  -> 发送 CHAT_MESSAGE 事件
```

### 4.6 推荐链路

```text
用户进入任务大厅
  -> recommendation_service 读取用户偏好与实时上下文
  -> map_service 计算距离得分
  -> credit_service 提供履约成功特征
  -> Redis 查询/刷新推荐缓存
  -> 返回推荐任务列表
```

## 5. WebSocket 设计

### 5.1 通道

| 通道 | 用途 |
|------|------|
| `notification:{userId}` | 个人通知 |
| `chat:{taskId}` | 任务内聊天 |
| `presence:{taskId}` | 会话在线状态与已读同步 |

### 5.2 事件

| 事件名 | 方向 | 用途 |
|------|------|------|
| `NOTIFICATION_CREATED` | 服务端 -> 客户端 | 新通知 |
| `TASK_STATUS_CHANGED` | 服务端 -> 客户端 | 状态更新 |
| `CHAT_MESSAGE` | 双向 | 新消息 |
| `CHAT_READ` | 双向 | 已读同步 |
| `UNREAD_SYNC` | 服务端 -> 客户端 | 未读数量同步 |
| `SYSTEM_NOTICE` | 服务端 -> 客户端 | 系统公告 |

### 5.3 鉴权与可靠性

- 连接时使用 `accessToken` 鉴权
- 断线重连时补拉离线未读
- 消息持久化成功后才广播
- 未读计数保存在 Redis，并周期性回写数据库

## 6. 地图与位置能力

### 6.1 底图方案

- 使用 OpenStreetMap 瓦片作为真实地图底图
- 前端基于 Leaflet 渲染缩放、拖拽和楼宇/任务覆盖层
- 后台维护楼宇编码、楼宇中心经纬度；有条件时补充楼宇轮廓多边形
- 前端将楼宇经纬度与任务点位映射到地图覆盖层

### 6.2 距离能力

- 首版按楼宇中心点做欧氏距离或曼哈顿距离估算
- 支持：
  - 附近任务排序
  - 推荐中的距离特征
  - 地图点聚合展示

### 6.3 不做的内容

- 不接入真实路径导航
- 不接入第三方地图导航 SDK 作为主方案

## 7. AI 审核架构

### 7.1 审核模式

- 发布前同步调用 DeepSeek API
- 结果分三类：
  - `ALLOW`
  - `REVIEW`
  - `BLOCK`

### 7.2 分级规则

| 级别 | 内容 | 系统行为 |
|------|------|------|
| 高危 | 暴力、色情、诈骗、违法 | 直接拦截 |
| 低危 | 辱骂、人身攻击、广告刷屏 | 允许创建任务，但进入管理员复审 |
| 正常 | 普通内容 | 正常通过 |

### 7.3 审核留痕

- 保存请求摘要、模型结果、标签、风险级别、管理员复审结论
- 管理员后台可查看与复审

## 8. 推荐系统架构

首版采用规则加权推荐，不引入复杂机器学习。

### 8.1 特征

| 特征 | 来源 |
|------|------|
| 类别偏好 | 历史接单类别 |
| 距离得分 | 当前楼宇与任务楼宇距离 |
| 历史完成成功率 | Helper 历史完成情况 |
| 近期活跃时段 | 近一周接单时段分布 |

### 8.2 处理流程

- 在线计算推荐得分
- 热门用户和热门条件使用 Redis 做短期缓存
- 后台允许调整各特征权重

## 10. 部署方案

### 10.1 生产部署

建议使用单台 Linux 云服务器，结合 Docker Compose 或等价容器方式部署：

```text
Nginx
  ├── 前端静态资源
  ├── FastAPI HTTP API
  └── FastAPI WebSocket

MySQL
Redis
```

对应仓库文件：

- `backend/Dockerfile`
- `frontend/Dockerfile`
- `docker-compose.prod.yml`
- `deploy/nginx.conf`
- `backend/.env.production.example`
- `frontend/.env.production.example`

这套方案的定位是：

- 本地开发：使用 `docker-compose.yml` 启动 MySQL 与 Redis，应用本地运行
- 生产/演示上线：使用 `docker-compose.prod.yml` 在单台 Linux 服务器统一编排前端、后端、MySQL、Redis 与 Nginx
- 当前课程项目推荐直接采用“单机自建 MySQL + Redis”方案，不把 RDS / 托管 Redis 作为默认前提

### 10.1.1 数据持久化

- MySQL 使用 Docker volume 持久化到服务器磁盘
- Redis 使用 AOF + Docker volume 做基础持久化
- 即使容器重建，挂载卷仍保留数据
- 该方案足以满足课程项目的演示与阶段验收

### 10.1.2 端口暴露策略

- 仅 Nginx 对外暴露 80 端口
- MySQL 与 Redis 不直接暴露公网端口
- 后端容器仅在内部网络中被 Nginx 访问

### 10.2 配置项

| 环境变量 | 说明 |
|------|------|
| `DATABASE_URL` | MySQL 连接串 |
| `REDIS_URL` | Redis 连接串 |
| `JWT_SECRET` | JWT 密钥 |
| `OSS_*` | OSS 相关密钥与配置 |
| `DEEPSEEK_API_KEY` | DeepSeek API 密钥 |
| `DEEPSEEK_BASE_URL` | DeepSeek API 地址 |
| `CORS_ORIGINS` | 前端域名白名单，逗号分隔 |

### 10.3 本地开发

- 前端：`npm run dev`
- 后端：`uvicorn`
- 数据：本地 MySQL + Redis
- WebSocket：与 FastAPI 同进程或同服务提供
- 本地数据库与 Redis 通过 `docker-compose.yml` 启动，并绑定到 `127.0.0.1`

## 11. 风险与约束

### 11.1 风险

- WebSocket 与 IM 增加部署复杂度
- 地图点位需要提前整理北邮楼宇数据
- DeepSeek 调用需要处理超时与额度
- 推荐规则过多会增加调参成本
- 单机部署虽然成本可控，但需要做好数据库卷持久化与备份

### 11.2 约束

- 所有关键业务规则必须先写入文档再开发
- 不允许开发阶段擅自更改状态机、信用分字段、事件名、核心接口
- `API接口规范.yaml` 为唯一接口基线
